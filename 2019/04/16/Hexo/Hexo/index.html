Hexo的思考

一个好的blog的标准

我为啥要做一个Hexo的博客？

一开始是因为配合github.io搭建可以直接架在公网上，便宜。但是随着我想增加一下Hexo的功能，比如写作时的热更新，主题的更换和魔改，关于Hexo的SEO等等，我发现Hexo并没有我想象中的那么方便。

我自己也写过一些简单的动态博客，用mongodb+express+react来构建。因为是动态博客，我自己也实现了markdown在线编辑，在线保存，实时预览的效果。这样看来Hexo在不装社区中大大小小的插件的时候，使用起来就不那么令人舒适。

和Hexo类似的，老牌的博客建站程序WordPress，它又有啥不同？

我想从博客这个产品的角度，web开发的角度来谈谈这三种blog

动态博客VS静态博客

前后端分离

假设你在运营一个论坛社区网站。一开始，你的用户很少，你不需要考虑很多复杂的问题。你是这么做的：

  • 选择一个云端服务商，购买一个云端虚拟服务器。
  • 在虚拟云端服务器上运行一个http服务器，提供路由，并将静态页面挂载到http服务器上。
  • 在虚拟云端服务器上运行一个数据库，http服务器根据路由，对数据库进行CURD，提供服务。

里面可能有着更多技术细节，这方面就不多赘述。总之我们可以一台服务器就能解决问题。

随着业务的发展，用户的增多，你开始有新的需求：希望不同地区，不同地域的人能够尽快的访问到你的页面。

这时候就有一个新的方案：上述的3个步骤都不会变化，仅仅是将静态页面托管到CDN上。大公司会有自己的CDN集群服务，也可以购买其他公司的CDN集群服务。

CDN简单来说，就是能让不同地方客户端的ip，能通过CDN服务器，访问到距离客户端最近（路由最近，时间最短）的服务器。

回到我们blog的话题，你就会发现：

  • WordPress

    • 定位CMS，动态站，可在线编辑文章，发布
    • 数据库+http服务器+页面，部署在一个服务器中
    • 可拓展性好

  • Hexo

    • 纯静态页面，无后端，可直接CDN部署
    • 更没有数据库,.md文件就是文章

  • 自己写的博客

    • 前后端分离，可以把静态页面部署到CDN
    • 定位CMS，动态站，可在线编辑文章，发布
    • 可拓展性差

Hexo是轻量级的静态站，和WordPress定位是完全不同的。WordPress是CMS，而且可拓展性很好。

我想下次谈Hugo时，仔细的比较一下WordPress。

用户体验

Hexo这种静态站，别人不能留言，真的成了静态展示站…

Hexo静态站，就只能本地编辑在deploy了，这个其实不算问题

何去何从

静态站可能写的就是一些md了，Hexo(github.io大法好)

动态站就会考虑使用Hugo了。(也可能真香了)(希望能和WordPress有很多不一样的地方)

自己写的博客可能才是最舒服的，维护起来自己也好维护。

社区开源CMS的思路好像只能适用于模板，不能配合react进行前后端分离吗？
